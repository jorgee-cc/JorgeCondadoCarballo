01 — Contexto

El Problema Real

Antes del sistema

  • Gestión de citas exclusivamente por teléfono
  • Doble reserva de horarios sin control
  • Sin trazabilidad ni historial por cliente
  • Dificultad para asociar vehículos a clientes
  • Cancelaciones de última hora incontroladas
  • Sin notificaciones automáticas al cliente

Después del sistema

  • Solicitudes online 24/7 con validación en tiempo real
  • Panel de administración centralizado con estados de cita
  • Historial completo de citas y vehículos por cliente
  • Asociación automática de matrículas al crear citas
  • Regla de cancelación de 24 horas aplicada automáticamente
  • Notificaciones vía email automáticas por eventos
02 — Arquitectura

Cómo Funciona el Sistema

Cliente Formulario Web
Backend Flask · Blueprint MVC
Base de Datos SQLite / PostgreSQL
Automatización n8n via Webhook
Notificación Email SMTP
⬡
Core
Monolito Flask con patrón Blueprint MVC

Tres módulos independientes: auth (autenticación), cliente (solicitudes y cancelaciones) y admin (gestión completa). Cada Blueprint aísla su lógica, rutas y decoradores de control de acceso por rol.

⊞
Outbox
Patrón Store-and-Forward para resiliencia

Si n8n no está disponible, Flask guarda la notificación en la tabla notificaciones_fallidas. Un Cron Job (Flask-APScheduler) reintenta cada hora hasta 24 veces antes de abandonar, evitando pérdida de datos.

▶
n8n
Motor de automatización vía Docker

Recibe webhooks de Flask con autenticación por cabecera. Un nodo Switch enruta según el tipo de evento (nueva_cita / estado_actualizado) y lanza el flujo de email correspondiente al admin y/o al cliente.

⛭
Infra
VPS Hetzner · Coolify · Traefik

Coolify gestiona contenedores Docker y certificados SSL. Traefik actúa como proxy inverso enrutando el tráfico a Flask (puerto 5000) y n8n (puerto 5678) sin exponerlos directamente. Ubuntu 24 endurecido con UFW, Fail2Ban y Snort IDS/IPS.

03 — Stack Tecnológico

Tecnologías Utilizadas

Backend
Python 3.12 · Flask

Framework web ligero con arquitectura de Blueprints para separación de módulos. ORM SQLAlchemy para consultas seguras parametrizadas.

Plantillas
Jinja2 · JavaScript ES6+

Motor de plantillas con auto-escape activo. AJAX con fetch API para actualizaciones dinámicas sin recarga de página completa.

Base de Datos
SQLite → PostgreSQL

SQLite para desarrollo; PostgreSQL como backend de producción para n8n. Diseño escalable con SQLAlchemy que no requiere cambios de código al migrar.

Automatización
n8n + Docker

Motor de workflows low-code desplegado como contenedor. Recibe webhooks autenticados, evalúa condiciones y ejecuta envíos SMTP.

Tareas programadas
Flask-APScheduler

Cron job integrado en la factoría create_app para reintentos automáticos de notificaciones fallidas con inyección manual de contexto de app.

Infraestructura
Hetzner VPS · Coolify 4

Panel self-hosted que gestiona Docker, SSL automático y deploy desde repositorio privado GitHub vía SSH Deploy Keys.

Proxy / Red
Traefik · UFW · Iptables

Traefik enruta tráfico SSL. UFW bloquea puertos no autorizados. Hashlimit en cadena DOCKER-USER como anti-DoS a nivel de red.

Seguridad
Fail2Ban · Snort IDS/IPS

Fail2Ban banea IPs tras intentos SSH fallidos. Snort detecta SQLi y XSS en el tráfico web y se integra con Fail2Ban para bans automáticos.

04 — Funcionalidades

31 Funcionalidades Implementadas

F-01 / F-02
Autenticación por roles

Registro y login con hash PBKDF2-SHA256 (600K iteraciones). Dos roles diferenciados: cliente y administrador con acceso completamente separado. La interfaz se adapta dinámicamente según el rol detectado vía AJAX.

F-03 / F-04 / F-05
Solicitud y gestión de citas

Reserva online con selección de fecha, hora, motivo y matrícula. Estados: pendiente → confirmada / cancelada → completada. Regla de cancelación de 24 horas aplicada automáticamente en backend y frontend.

F-06 — F-13
Panel de administración de citas

Vista completa con filtrado por fecha. El admin acepta, rechaza, completa o modifica manualmente el estado de cualquier cita. Puede añadir observaciones internas invisibles para el cliente.

F-14 — F-23
Gestión completa de usuarios

CRUD completo con verificación de contraseña del admin para acciones sensibles. Búsqueda por nombre y matrícula. Cambio de roles, historial de citas por cliente separado en activas y completadas.

F-24 — F-26
Gestión de vehículos (N:M)

Relación muchos-a-muchos entre usuarios y vehículos. Un vehículo puede pertenecer a varios usuarios (coche familiar). La eliminación de un usuario no borra el vehículo si está compartido.

Sistema
Notificaciones automáticas

Envío asíncrono via threading para no bloquear el hilo principal de Flask. n8n enruta por tipo de evento y envía emails HTML personalizados al admin y al cliente mediante SMTP.

06 — Ingeniería

Retos y Soluciones

Reto

Envíos de email bloqueando el servidor

Los envíos síncronos a n8n bloqueaban el hilo principal de Flask causando latencia visible en la UI. Además, si n8n caía temporalmente, la notificación se perdía definitivamente.

Solución

Threading + Patrón Outbox

El envío se delega a un hilo secundario con threading.Thread. Si falla, se guarda en notificaciones_fallidas y APScheduler reintenta cada hora hasta 24 veces. Zero pérdida de datos.

Reto

CSP estricta rompiendo toda la interfaz

Al aplicar CSP sin unsafe-inline para prevenir XSS, el navegador bloqueaba todos los manejadores de eventos onclick="..." usados extensivamente en los paneles de admin.

Solución

Delegación de eventos + atributos data-*

Eliminación de todos los onclick del HTML. Event listeners en contenedor padre leyendo atributos data-cita-id, data-estado. CSP con nonce único por petición en su lugar.

Reto

CSRF completamente inoperante

El sistema creía tener CSRF activo, pero un bug de precedencia de operadores en Python hacía que la validación nunca se ejecutara realmente, dejando todos los formularios POST desprotegidos.

Solución

HMAC-SHA256 + csrfFetch

Reescritura completa del sistema CSRF con tokens HMAC-SHA256. Función csrfFetch estandarizada en todo el frontend que inyecta automáticamente el token en cada petición POST/PATCH/DELETE.

Reto

IP Spoofing evitando el rate limiter

El rate limiter leía directamente la cabecera X-Forwarded-For sin validar, permitiendo a atacantes cambiar su IP aparente en cada request y evadir completamente el sistema anti-DoS.

Solución

ProxyFix con num_proxies=1

Middleware de Werkzeug configurado para confiar solo en el proxy legítimo (Traefik). La IP del cliente se normaliza correctamente antes de aplicar rate limiting, invalidando el vector de spoofing.

Mejoras Futuras

Celery / Redis Queue

Envío de presupuestos en PDF directamente adjuntos al email del cliente vía Multipart/form-data en n8n.
